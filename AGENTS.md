# AGENTS.md

## 기본 작업 규칙

- 작업 전 `AGENTS.md`, `프로젝트_기획서.md`, `개발_로드맵.md`, `다음_작업.md`, `작업요약.md`를 먼저 확인한다.
- `작업기록/` 폴더가 있으면 최신 작업기록 파일을 확인한다.
- 작업 대상 폴더인 `01-my-legend-app` 안에서만 파일을 수정한다.
- `MyPractice` 안의 다른 프로젝트 폴더는 수정하지 않는다.
- 사용자가 요청한 작업 범위를 넘지 않는다.
- 기존 기능, 화면 구조, 문서 기준을 함부로 변경하지 않는다.
- 불필요한 리팩토링을 하지 않는다.
- 요구사항이 불명확하면 임의로 확정하지 않고 `추후 결정 필요`라고 표시한다.
- 기능 추가 또는 변경 후 관련 문서를 함께 갱신한다.

## Next.js 작업 규칙

- 이 프로젝트는 Next.js 기반이다.
- Next.js 관련 코드를 수정하기 전 `node_modules/next/dist/docs/`의 관련 문서를 우선 확인한다.
- App Router, Client Component, Metadata 등 버전별 규칙을 확인하고 작업한다.
- 인터랙션, 상태, 브라우저 API가 필요한 컴포넌트에는 `"use client"`를 사용한다.

## 제품 방향 규칙

- 이 프로젝트는 단순 랜덤 추첨 앱이 아니다.
- 목표는 `관전형 랜덤 레이스 게임`이다.
- 사용자는 결과만 보는 것이 아니라 결과가 만들어지는 과정을 즐겨야 한다.
- 결과 조작, 특정 참가자 우대, 고정 승리는 금지한다.
- 랜덤성은 공정해야 하며, 예측 불가능한 재미를 만들어야 한다.

## 개발 규칙

- MVP를 먼저 완성한다.
- 기능은 작은 단위로 구현한다.
- 한 번에 여러 기능을 무리하게 추가하지 않는다.
- 구현 후 즉시 동작 확인이 가능한 상태를 유지한다.
- 오류 발생 시 원인을 분석하고 수정한다.
- 임시 코드가 남지 않도록 한다.
- 하드코딩을 최소화한다.
- 반복적으로 조정되는 값은 상수 또는 설정값으로 분리한다.
- 물리, 카메라, 맵, UI 로직은 가능하면 역할별로 분리한다.

## UI 규칙

- 사용자가 누가 앞서는지 쉽게 이해할 수 있어야 한다.
- 참가자 이름은 멀리서도 식별 가능해야 한다.
- 레이스 화면을 과도한 UI가 가리지 않도록 한다.
- 모바일과 PC 환경을 모두 고려한다.
- 화려함보다 가독성과 관전 흐름을 우선한다.
- 순위판, 이벤트 표시, 결과 화면은 직관적으로 구성한다.

## 문서 관리 규칙

### `프로젝트_기획서.md`

- 프로젝트의 최종 기준 문서로 사용한다.
- 목표, 사용자, 핵심 기능, 화면, 데이터, 권한, 성공 기준을 관리한다.
- 기능 추가 또는 변경 시 최신 상태로 유지한다.

### `개발_로드맵.md`

- MVP부터 장기 확장까지 단계별 개발 계획을 관리한다.
- 완료된 항목은 체크 처리한다.
- 우선순위가 바뀌면 변경 사유를 기록한다.

### `다음_작업.md`

- 다음 작업에서 바로 수행할 작업만 기록한다.
- 완료된 작업은 제거하고 다음 작업으로 갱신한다.
- 여러 스프린트를 동시에 적지 않는다.

### `작업요약.md`

- 지금까지 진행 상황을 요약한다.
- 새 작업 완료 시 반드시 갱신한다.
- 현재 상태, 완료된 작업, 남은 작업, 주의사항을 기록한다.

### `작업기록/`

- 날짜별 작업 기록을 저장한다.
- 파일명 예시: `작업기록/2026-06-07.md`
- 작업 내용, 수정 파일, 주요 결정 사항, 문제 및 해결 내용을 기록한다.

## 코드 품질 규칙

- 가독성을 우선한다.
- 함수는 하나의 역할만 담당하도록 작성한다.
- 중복 코드를 최소화한다.
- 파일 구조를 단순하게 유지한다.
- 주석은 필요한 경우에만 작성한다.
- 테스트 또는 검증 방법을 작업 결과에 포함한다.
